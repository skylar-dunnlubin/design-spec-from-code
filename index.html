The Problem

Traditional DHF workflows assume design precedes code. Agentic coding inverts that.

Medical device development follows a well-established waterfall for design controls: User Needs → Design Inputs → Design Specification → Code → Verification. The Design Specification — traditionally a Figma file, annotated mockups, or a formal document — serves as the bridge between what was designed and what gets implemented. It's the artifact an auditor traces from user needs to code.

When you "vibe code" a feature with Claude Code, the code comes first. The design decisions — layout choices, component structures, interaction patterns, error handling — are made in real-time during the coding process. They exist in the codebase, but nowhere else. No Figma file. No annotated wireframes. No formal specification document.

This creates a traceability gap in your Design History File. When an auditor asks "Show me the design specification for this feature," you have nothing to point to — even though every design decision was intentional and the feature works correctly.

The gap isn't about quality. The coded feature may be excellent — thoughtfully designed, well-structured, thoroughly tested. The gap is purely documentary. The design decisions exist, they just aren't captured in the format your quality system requires.

The Hypothesis

The design decisions are in the code. They just need extraction.

When you build a feature with Claude Code, you're not writing random code. You're making deliberate design decisions: how components are structured, what layout patterns to use, how data flows to the UI, what error states to handle, how interactions work. These decisions are the design specification — they're just expressed in code rather than in a Figma file.

Claude Code can read a codebase and identify every design decision embedded in it: component hierarchies, visual patterns, interaction flows, data display rules, accessibility implementations, responsive behaviors. If those decisions can be extracted, structured, and documented, the result is a design specification that is more accurate than a traditional pre-implementation spec.

A spec generated from code describes what was actually built. A traditional spec describes what was intended. In practice, every production codebase diverges from its original design spec — developers make pragmatic decisions during implementation that never make it back to Figma. Code-first, spec-second eliminates this drift entirely.

04

Regulatory Foundation

Where the design specification sits in the Design History File — and why temporal order doesn't matter.

FDA

21 CFR 820.30

Design Controls. Requires design inputs, design outputs, design verification, design validation, and a Design History File. The Design Specification is a design output — it defines the design solution in terms that allow evaluation against design inputs.

§820.30(d) "Each manufacturer shall establish and maintain procedures for defining and documenting design output in terms that allow an adequate evaluation of conformance to design input requirements."

The regulation requires conformance evaluation — that the spec shows the design meets inputs. It does not specify temporal ordering of when the spec is created relative to implementation.

IEC

IEC 62304

Software Lifecycle Processes. Requires Software Architectural Design documentation (decomposing the system into components and interfaces) and Software Detailed Design documentation (logic, data flow, interfaces for each unit). These map directly to what Claude Code can extract from a codebase.

IEC

IEC 62366

Usability Engineering. Requires documentation of user interface specifications, including interaction patterns, visual design decisions, and accessibility considerations. The extracted design spec captures all of these from the actual implementation.

Design Controls V-Model — Where the Design Spec Sits
The critical regulatory insight: Standards require the design specification to exist and be traceable. They require it to demonstrate conformance to design inputs. They do not mandate that it must be authored before implementation begins. The traceability chain — user needs → design inputs → design spec → code — holds regardless of temporal ordering.

Objections & Risk Framing

Addressing the concerns your quality team will raise — head-on.

Objection "The design specification should come before implementation."
Response

Regulatory standards require the spec to exist and be traceable, not that it precedes code temporally. FDA 820.30(d) requires design output to allow "adequate evaluation of conformance to design input requirements." A spec extracted from code and validated against design inputs satisfies this requirement — and more accurately, since it reflects what was actually built.

Objection "This is just retroactive documentation — writing the spec after the fact."
Response

It's extraction + review, not retroactive justification. The code was written with intentional design decisions. Claude Code extracts and structures those decisions into a formal document. A designer/QA professional then reviews the spec against design inputs and signs off. This review step is the same review that happens in traditional workflows — it just occurs after extraction rather than after handoff.

Objection "An auditor won't accept a design spec generated from code."
Response

Auditors verify traceability, not process ordering. If the spec clearly traces back to design inputs, was formally reviewed and approved, and maps to specific code components — the traceability chain is complete. The auditor sees: user needs → design inputs → reviewed design spec → verified code. The DHF is intact.

Objection "What about design reviews? Don't those need to happen before coding?"
Response

Design reviews happen after the spec is generated and before it's approved. The designer reviews visual consistency, interaction completeness, and accessibility. QA reviews traceability to design inputs. This is substantively the same review — it just reviews the actual implementation rather than an aspirational mockup.

Objection "Spec-code drift is a feature — it helps catch implementation errors."
Response

In practice, spec-code drift almost always means the spec is outdated, not that the code is wrong. Developers make pragmatic decisions during implementation that improve the design but never make it back to Figma. This approach eliminates that ambiguity entirely — the spec always matches the code because it's derived from it. Errors are caught during the review step, where the spec is validated against design inputs.

Objection "How do we track what the AI changed vs. what a human wrote?"
Response

Git tracks every change regardless of origin. Co-Authored-By headers in commits make AI involvement transparent. The human who invoked the tool owns the change — same as using any other development tool. The segregation comes from review, not authorship.

Objection "AI-generated code needs different governance than human-written code."
Response

It doesn't. Your existing process — PR reviews, Jira traceability, CI gates, 2-reviewer approval — governs the output, not the method of production. Treat AI-assisted code identically. The only addition: document Claude Code in your tool register (one-time qualification) and ensure Co-Authored-By headers for transparency.

Objection "What stops an engineer from modifying design-verified UI and merging without design review?"
Response

Nothing stops it technically — and that's true with or without this workflow. What this approach adds is visibility. Every spec carries a verification status tied to the merge commit on main. When UI code changes post-verification, the spec goes STALE. The changelog shows exactly what diverged, when, and by whom. You can't prevent unauthorized changes, but you can make them impossible to hide. Pair with a CI check that flags PRs touching UI code without corresponding spec updates.

The Methodology

Seven steps from code to audit-ready, verification-ready design specification — including ongoing maintenance.

1

Code the Feature

Build the feature using Claude Code. Iterate on the UI, interactions, and data display until it works correctly and meets the design intent. This is your normal agentic coding workflow — focus on building a great feature.

Your existing workflow — no change needed
2

Analyze the Codebase

Prompt Claude Code to read and map the codebase: component tree, screen inventory, navigation paths, shared utilities, design system usage, data models. This produces a structural understanding of what was built.

Claude Code reads your code and builds a mental model
3

Extract Design Decisions

Prompt Claude Code to identify the design decisions embedded in the code: layout patterns, component states, interaction behaviors, error handling, visual design choices, accessibility implementations. Each decision gets documented with its rationale.

Design decisions become explicit & documented
4

Generate the Specification

Claude Code outputs a structured design specification document following the template (see Section 08). The spec includes all 12 required sections: from document header through verification summary. Each section references specific code files and components, with acceptance criteria built into every component and flow.

Structured document ready for review
5

Review & Approve

A designer reviews visual consistency, interaction completeness, and design system alignment. QA reviews traceability to design inputs and regulatory compliance. Both sign off on the spec. This is your standard design review — same rigor, same approvals.

Same review process as traditional workflow
6

Lock Traceability

The approved spec is version-controlled alongside the code. Code files reference the spec version. The spec references specific code files and line ranges. The traceability chain is complete: user needs → design inputs → design spec → code. The DHF entry is documented.

Traceability chain complete — DHF is audit-ready
The spec is now a verification artifact. Any team member — designer, engineer, QA, or AI — can read the spec's acceptance criteria (§4, §6, §7) and verification summary (§12) to systematically verify the production code meets every requirement.
7

Maintain the Spec

The spec is a living document. Three scenarios trigger maintenance:

  • Code changes an existing feature — Run the Update prompt → delta + changelog + STALE status. Designer re-reviews. If approved: re-verify. If unauthorized: flag for discussion.
  • New feature added to product — Run Extract for new feature → new spec. Update architecture doc (add to Feature Inventory). Run Update on impacted existing specs.
  • Feature removed — Update architecture doc (remove from inventory). Archive the feature spec. Changelog entry documenting removal.
Living process — Extract + Update cover every scenario

Post-Handoff Integrity: The Changelog Model

The core innovation addressing the deepest engineering concern: what happens after design hands off code?

The problem: After design verifies and hands off UI code, engineers can modify it in subsequent PRs with only engineer approval. The design-verified state is silently overwritten.

Top of every spec and sub-spec:

How It Works in Practice

1 Design extracts spec → changelog entry + verification status set to VERIFIED
2 Engineer modifies UI code → their merged PR should update the spec (or CI flags it)
3 If spec is updated → changelog entry shows who changed what and why
4 If spec is NOT updated → verification status goes STALE → visible to design + QA
5 Design reviews STALE specs periodically → re-verifies or raises concerns
You can't prevent engineers from modifying UI code. But you can make it impossible to do so silently. The changelog is the audit trail. Divergence becomes a conversation — not a silent override.

What Goes in the Design Spec

The 12 sections that make a design specification audit-ready and verification-ready — and why each matters.

1

Document Header

Version, date, author, feature scope, applicable regulatory standards (820.30, 62304, 62366). Establishes the document's identity and regulatory context.

Why: Auditors need to identify the document, its version, and which regulations it addresses.
2

Design Intent

What the feature does, which user needs it addresses, and explicit links to design input documents. This is the traceability anchor — every other section traces back to here.

Why: Establishes the chain from user needs → design inputs → this specification.
3

Screen Inventory

Every screen or view in the feature: name, purpose, navigation path, parent/child relationships. Include screenshots or rendered views where possible.

Why: Provides the map of the UI surface area. An auditor can verify every screen was designed intentionally.
4

Component Specification

Each UI component: visual description, all states (default, hover, active, disabled, error, loading), behavior, props/parameters, and accessibility attributes.

Why: Components are the building blocks. Documenting their states ensures no interaction was accidental.
5

Layout & Visual Design

Grid system, spacing scale, typography hierarchy, color usage, responsive breakpoints, and how the layout adapts across viewport sizes.

Why: Visual design decisions are design outputs. Layout rules demonstrate systematic, not arbitrary, design.
6

Interaction Design

User flows through the feature, transitions between states, error state handling, loading behaviors, edge cases, and gesture/input handling.

Why: Interactions define the user experience. Documenting them shows every flow was considered.
7

Data Display

How data maps to UI elements: formatting rules, units, precision, empty states, error states, real-time update behavior, and data freshness indicators.

Why: Medical devices display critical data. Every formatting decision has clinical implications.
8

Design Rationale

Why specific design choices were made: trade-offs considered, alternatives evaluated, constraints that drove decisions. Links back to user needs and design inputs.

Why: The traceability core. An auditor understands HOW design inputs became design decisions.
9

Design System Alignment

Which design tokens, patterns, and components from the existing design system were used. Deviations from the design system and their justifications.

Why: Demonstrates consistency with the broader product design language and documents exceptions.
10

Accessibility

WCAG compliance level, contrast ratios, screen reader behavior, keyboard navigation, touch target sizes, and any accessibility-specific design decisions.

Why: IEC 62366 usability requirements. Accessibility is a design output that must be documented.
11

Traceability Matrix

A table mapping: user needs → design input references → spec section numbers → code file paths and component names. This is the auditor's roadmap.

Why: The single artifact that proves end-to-end traceability. Auditors start here.
12

Verification Summary

Consolidated pass/fail checklists: component states to verify, flow entry/exit and error paths, data validation rules and boundary tests, traceability verification status. The test oracle for the entire feature.

Why: Makes the spec usable as a verification oracle. A human or AI can read this section and systematically determine whether the code is correct.

For Developers

How to code review against a generated design spec — and how to trust it.

In this workflow, you receive code alongside a design specification that was generated from that code. Your role shifts: instead of implementing a spec, you're verifying that a spec accurately captures what the code does. This is a different kind of code review — and in many ways, a more valuable one.

Code Review Workflow

1
Open the PR + the design spec side by side. The spec should reference specific files and components. Use the traceability matrix (Section 11) as your review map.
2
Verify component states. For each component in the spec, check that the code handles every listed state: default, hover, active, disabled, error, loading. Flag any states the spec lists that the code doesn't handle — or vice versa.
3
Verify interactions. Walk through each user flow described in the spec. Does the code implement every transition, error path, and edge case? Are loading states handled?
4
Verify data display. Check formatting rules, units, precision, and empty/error states against the spec. Medical data display is safety-critical — this section deserves extra scrutiny.
5
Flag discrepancies. If the spec says one thing and the code does another, the code is the source of truth in this workflow. Flag the discrepancy so the spec gets updated — never change the code to match a generated spec.
The key mental model shift: In traditional workflows, code must match the spec. In this workflow, the spec must match the code. When they disagree, update the spec. The code was built intentionally — the spec is a description of that intent, and descriptions can be imperfect.

How to Trust the Generated Spec

What to verify

  • Component states match actual code behavior
  • User flows cover all paths (happy + error)
  • Data formatting rules match implementation
  • Accessibility claims are real (test with screen reader)
  • Traceability matrix points to correct files

What to flag

  • States described in spec but not implemented
  • Code behavior not captured in spec
  • Vague design rationale (needs specificity)
  • Missing error/edge case documentation
  • Broken traceability links (file paths changed)

For Designers

Your role evolves — from author to reviewer, validator, and design system steward.

This workflow doesn't replace designers. It changes when you engage. Instead of creating mockups that become the spec, you review an extracted spec against design inputs and validate that the implemented design meets user needs, follows design principles, and maintains system consistency.

Reviewer

Primary

Review the generated design spec for visual consistency, interaction completeness, and user experience quality. You're the quality gate between "code that works" and "design that's right."

  • Does the layout follow established patterns?
  • Are interactions intuitive and complete?
  • Do error states guide the user effectively?
  • Is the information hierarchy correct?

Validator

Critical

Validate that the design meets user needs and design inputs. This is the regulatory traceability check — does what was built actually address what was intended?

  • Does the feature address the documented user needs?
  • Are design inputs satisfied by the implementation?
  • Would a user accomplish their task effectively?
  • Are there accessibility gaps?

Design System Steward

Ongoing

Maintain and evolve the design system that guides future agentic coding. The stronger the design system, the better Claude Code's output. Your design system becomes the "design input" for code generation.

  • Update tokens and patterns based on reviewed specs
  • Document new components that emerge from coding
  • Ensure consistency across features
  • Define interaction patterns for future features

When to Intervene

Flag the spec for revision when:

  • The design doesn't meet user needs — even if it technically works
  • Visual design violates established design principles or brand guidelines
  • Interaction patterns are inconsistent with the rest of the product
  • Accessibility falls below your standards
  • The design rationale is weak or missing — decisions should be traceable to user needs

Worked Examples

Two scenarios showing the methodology end-to-end.

Scenario

You've built an Admin Hub from scratch using Claude Code — a web dashboard for configuring patient monitoring settings, managing sensor assignments, and setting alert thresholds. The codebase is complete and working. No design spec exists. You need to generate one for the DHF.

Step 1 — Codebase Analysis Prompt
Read the entire Admin Hub codebase at /src. Map the following:
1. Every screen/page: name, route, purpose
2. Component tree: parent-child relationships
3. Shared components and utilities
4. Data models and API endpoints consumed
5. Navigation structure and user flows
6. Design system usage: tokens, theme, shared styles

Output a structured codebase analysis document.
Step 2 — Design Decision Extraction
Using the codebase analysis, extract every design decision:
1. Layout patterns: grid systems, responsive behavior, page structure
2. Component states: all states each component handles
3. Interaction patterns: forms, modals, navigation, real-time updates
4. Visual design: colors, typography, spacing, iconography
5. Data display: formatting rules, units, empty/error states
6. Error handling: validation, error messages, recovery flows
7. Accessibility: ARIA usage, keyboard nav, contrast, screen reader support

For each decision, note the rationale (why this approach vs alternatives)
and the code file where it's implemented.
Step 3 — Spec Generation
Generate a complete Design Specification document following the
11-section template. The document must:
- Reference specific code files for every component and screen
- Include all component states discovered in the extraction
- Document every user flow with error paths
- Map to these design inputs: [DI-001: ..., DI-002: ...]
- Include a traceability matrix linking user needs to code files
- Use formal regulatory language suitable for a DHF

Output as a single Markdown document.
Result

Claude Code produces a complete design specification covering the Admin Hub's 8 screens, 24 components, 12 user flows, and full traceability matrix. The designer reviews for visual consistency, QA reviews for traceability. After one round of revisions (two missing error states, one incomplete flow), the spec is approved and enters the DHF.

Scenario

You've added vitals monitoring features to an existing Mobile Hub app using Claude Code — new Vitals Panel, Alert History, Clinical Notes, and a redesigned Patient Detail screen. An existing design spec covers the original app. You need a feature addition spec for the DHF.

Step 1 — Codebase Analysis Prompt
Read the Mobile Hub codebase. I've added new features. Identify:
1. NEW screens/composables not in the original spec
2. MODIFIED screens/composables that existed before
3. New shared components and utilities added
4. New data models and Firebase paths consumed
5. Changes to navigation structure
6. New design tokens or theme changes

Compare against the existing spec at [path] and document
only what's new or changed.
Step 2 — Design Decision Extraction
For only the NEW and MODIFIED code, extract design decisions:
1. New component patterns introduced (VitalCard, AlertItem, etc.)
2. New interaction patterns (real-time updates, alert modals, etc.)
3. Changes to existing patterns (Patient Detail layout changes)
4. New data display rules (vital signs formatting, alert severity)
5. New error/edge cases handled
6. Impact on existing functionality

For modified components, document BEFORE and AFTER behavior.
Step 3 — Spec Generation
Generate a Design Specification Addendum following the brownfield
template. The addendum must:
- Reference the parent spec: DS-MH-BASE-002
- Clearly distinguish NEW vs MODIFIED vs UNCHANGED
- Document new components with full state specifications
- Document modified components with before/after behavior
- Include delta traceability matrix
- Include impact assessment for changes to existing functionality
- Map to new design inputs: [DI-050: vitals monitoring, ...]

Output as a single Markdown document.
Result

Claude Code produces a specification addendum documenting 4 new screens, 15 new components, 8 modified components, and 6 new user flows. The delta traceability matrix maps new features to new design inputs while confirming existing traceability is preserved. After review, the addendum is approved and filed alongside the parent spec in the DHF.
